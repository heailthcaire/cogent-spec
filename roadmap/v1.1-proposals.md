# Cogent v1.1 Proposals

This file tracks non-breaking improvements proposed for Cogent v1.1.

## 1. Per-Iteration Error Handling

### Problem

In v1.0, a failed `REQUIRE` inside `FOR EACH` or `PARALLEL FOR EACH` aborts the entire loop unless the iteration body is wrapped in `TRY/CATCH`.

Sometimes this is correct. Often, a workflow should skip or mark the failed item and continue.

### Possible Syntax

```text
PARALLEL FOR EACH competitor IN competitors_found:
  CONTINUE ON ERROR:
    score = CALL METHOD score-competitor[[competitor]]
    Append score to scored.
```

Alternative:

```text
PARALLEL FOR EACH competitor IN competitors_found:
  TRY ITEM:
    score = CALL METHOD score-competitor[[competitor]]
    Append score to scored.
  CATCH err:
    Append { competitor, error: err.message } to failed.
```

### Open Questions

- Should the keyword be `CONTINUE ON ERROR`, `TRY ITEM`, or normal `TRY/CATCH` inside the loop?
- Should failed items be automatically collected?
- Should `PARALLEL FOR EACH` allow item-level recovery without mutating shared state?

## 2. Typed Parameters

### Problem

v1.0 declares parameter names but not parameter types.

### Possible Syntax

```text
DEFINE METHOD score-competitor([[competitor_name: string]]):
  ...
```

```text
DEFINE METHOD compare-options([[options: array]], [[criteria: object]]):
  ...
```

### Potential Types

- `string`
- `number`
- `boolean`
- `array`
- `object`
- named object shapes

### Open Questions

- Should types be optional everywhere?
- Should shape declarations live in `exports` only, or also in method definitions?
- Should natural-language type descriptions be allowed?

## 3. Runtime Support for Parallel Execution

### Problem

`PARALLEL FOR EACH` is currently a semantic contract, not a guaranteed execution behavior.

### Proposal

When runtimes support true concurrent execution, `PARALLEL FOR EACH` should allow independent iterations to run concurrently.

### Requirements

- Iterations must not mutate shared state.
- Runtime should preserve deterministic final ordering when possible.
- Runtime should define error aggregation behavior.

## 4. Standard Error Object

### Problem

v1.0 refers to `err.message` but does not define a standard error shape.

### Possible Shape

```text
{
  code: string,
  message: string,
  source_method: string,
  failed_requirement: string,
  recoverable: boolean
}
```

### Open Questions

- Which fields are required?
- Should all failures be recoverable by default inside `TRY/CATCH`?
- Should missing imports and shape mismatches have standardized error codes?

## 5. Conformance Validator Format

### Problem

The v1.0 spec says Cogent skills are statically verifiable, but it does not define a validator output format.

### Possible Output

```json
{
  "valid": false,
  "cogentVersion": "1.0",
  "errors": [
    {
      "code": "MISSING_EXPORT_RETURN_SHAPE",
      "line": 8,
      "message": "Exported method main must declare a return shape."
    }
  ],
  "warnings": []
}
```

### Open Questions

- Should validator errors be standardized in the spec?
- Should warnings and errors be separated?
- Should validation include semantic checks or only syntax/shape checks?

## 6. Prompt-to-Cogent Translation Guidance

### Problem

Users will often begin with long, unstructured prompts. v1.1 may need guidance for converting rambling prompts into Cogent-compliant methods and workflows.

### Potential Guidance

- Extract goal.
- Identify reusable methods.
- Identify dependencies.
- Define inputs and outputs.
- Add `REQUIRE` postconditions.
- Add `TRY/CATCH` for likely failure points.
- Add conformance checklist.

### Open Questions

- Should this be part of the spec or a companion guide?
- Should examples include before/after prompt rewrites?
- Should an official online tool support this transformation?
